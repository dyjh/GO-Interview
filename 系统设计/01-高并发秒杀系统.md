# 高并发秒杀系统

## 秒杀系统的核心挑战

秒杀不是单纯加分布式锁或队列就能解决。核心挑战包括：

- 瞬时流量远高于日常流量。
- 库存有限，不能超卖。
- 用户请求大量重复点击。
- 数据库写入压力大。
- 下游服务不能被打穿。
- 用户体验需要明确反馈。

设计目标：

- 不超卖。
- 尽量少卖。
- 主链路高可用。
- 请求可削峰。
- 失败可补偿。
- 数据最终一致。

## 总体架构

```text
用户
 -> CDN/静态页
 -> 网关/Nginx 限流
 -> 秒杀服务
 -> Redis 预扣库存
 -> MQ 削峰
 -> 订单服务/库存服务
 -> MySQL 落库
```

## 前端和接入层

### 页面静态化

秒杀活动页、商品信息、规则等尽量静态化，放到 CDN，减少动态服务压力。

### 防重复点击

前端按钮置灰只能改善体验，不能作为安全保证。后端仍要做幂等和限流。

### 网关限流

在 Nginx/API 网关层按 IP、用户 ID、活动 ID 限流，把明显异常流量挡在应用前面。

### 风控

常见策略：

- 登录校验。
- 验证码或滑块。
- 黑名单。
- 设备指纹。
- 请求签名。
- 活动链接动态化。

## 业务层设计

### 库存预热

活动开始前把库存加载到 Redis：

```text
seckill:stock:{goods_id} = 1000
```

### Redis 原子预扣

使用 Lua 保证“判断库存 + 扣减”原子：

```lua
local stock = tonumber(redis.call("GET", KEYS[1]))
if stock == nil or stock <= 0 then
    return 0
end
redis.call("DECR", KEYS[1])
return 1
```

预扣成功只代表拿到资格，不代表最终下单成功。

### 一人一单

使用 Redis set 或数据库唯一索引：

```text
seckill:user:{activity_id}
```

Lua 中可同时判断用户是否已抢：

```lua
if redis.call("SISMEMBER", KEYS[2], ARGV[1]) == 1 then
    return -1
end
local stock = tonumber(redis.call("GET", KEYS[1]))
if stock == nil or stock <= 0 then
    return 0
end
redis.call("DECR", KEYS[1])
redis.call("SADD", KEYS[2], ARGV[1])
return 1
```

数据库层仍要用唯一索引兜底：

```sql
UNIQUE KEY uk_activity_user(activity_id, user_id)
```

## MQ 削峰

Redis 预扣成功后发送下单消息到 MQ：

```text
CreateSeckillOrder(activity_id, goods_id, user_id, request_id)
```

消费者按数据库承载能力消费，避免请求直接打到 MySQL。

注意：

- 消息要可靠投递。
- 消费者要幂等。
- 失败要重试和补偿。
- 消息堆积要监控。

## 数据库层

订单创建和库存扣减要在事务中完成：

```sql
UPDATE goods_stock
SET stock = stock - 1
WHERE goods_id = ? AND stock > 0;
```

如果影响行数为 1，再创建订单。

优化：

- 库存表按活动/商品拆分。
- 订单表分库分表。
- 使用唯一索引防重复下单。
- 热点商品库存可分段库存，减少单行热点。

## 防超卖

多层保证：

1. Redis 原子预扣。
2. MQ 串行或限速消费。
3. 数据库 `stock > 0` 条件扣减。
4. 唯一索引防重复下单。
5. 对账补偿。

不要只依赖应用内锁，因为多实例部署下本地锁无效。

## 少卖问题

少卖原因：

- Redis 预扣成功后用户未支付。
- 消息消费失败。
- 数据库落库失败。
- 用户重复请求占用资格。

解决：

- 订单超时未支付释放库存。
- 消费失败补偿库存。
- 定时对账 Redis、订单和库存。
- 活动结束后以数据库最终结果为准。

## 限流、降级和熔断

限流：

- 网关限流。
- 用户限流。
- 商品限流。
- Redis 令牌桶。

降级：

- 非核心服务如推荐、优惠券、积分延后处理。
- 查询接口返回缓存。

熔断：

- 下游数据库或 MQ 异常时，快速失败，保护系统。

## 用户体验

秒杀请求不要长时间阻塞等待最终订单结果。

可返回：

- 抢购资格处理中。
- 排队中。
- 抢购成功，待支付。
- 已售罄。
- 重复参与。

前端轮询或 WebSocket 查询订单状态。

## 监控指标

- QPS。
- 限流数量。
- Redis 扣减成功/失败。
- MQ 堆积。
- 下单成功率。
- 数据库 update 影响行数。
- 重复下单拦截数。
- 订单超时取消数。
- Redis 与 DB 库存差异。

## 面试追问

### 只用分布式锁能做秒杀吗？

不建议。分布式锁会把高并发串行化，吞吐低且锁竞争严重。秒杀更常用 Redis 原子扣减、队列削峰、数据库兜底和异步补偿。

### 如何保证不重复下单？

Redis 先拦截一人一单，数据库唯一索引最终兜底，消费者用 request_id 或业务唯一键保证幂等。

### Redis 挂了怎么办？

可提前限流降级，活动短暂不可用或切换备用 Redis。核心库存最终以数据库为准，但直接打数据库要严格限流，避免雪崩。
