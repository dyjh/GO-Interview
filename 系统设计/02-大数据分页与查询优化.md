# 大数据分页与查询优化

## 大分页为什么慢

传统分页：

```sql
SELECT *
FROM orders
ORDER BY id
LIMIT 1000000, 20;
```

问题：

- MySQL 需要扫描并丢弃前 1000000 行。
- 如果查询字段不在索引中，还会大量回表。
- 排序和临时表成本高。
- 页码越深越慢。

## 优化方向

### 1. 使用覆盖索引减少回表

先用索引查出 ID，再回表查详情：

```sql
SELECT o.*
FROM orders o
JOIN (
    SELECT id
    FROM orders
    WHERE status = 1
    ORDER BY id
    LIMIT 1000000, 20
) t ON o.id = t.id;
```

这仍然要跳过很多索引记录，但比直接回表扫描大量行更好。

### 2. Seek Pagination

也叫游标分页或基于上一页最后一条记录分页：

```sql
SELECT *
FROM orders
WHERE id > 1000000
ORDER BY id
LIMIT 20;
```

适合下一页/上一页，不适合随机跳页。

如果按时间排序：

```sql
SELECT *
FROM orders
WHERE (created_at, id) < (?, ?)
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

需要联合索引：

```sql
CREATE INDEX idx_created_id ON orders(created_at, id);
```

### 3. 限制最大页数

很多业务不需要无限深分页。例如搜索结果只允许看前 100 页，深层数据通过筛选条件缩小范围。

### 4. 冷热数据分离

近期热数据保留在主表，历史数据归档到历史表或 OLAP。

### 5. 使用搜索或分析系统

复杂筛选、排序、全文检索可同步到 Elasticsearch。大规模聚合分析可用 ClickHouse、Doris、StarRocks 等。

## 分页总数问题

```sql
SELECT COUNT(*) FROM big_table WHERE ...
```

在大表复杂条件下可能很慢。

优化：

- 不展示精确总数。
- 展示“更多”而不是总页数。
- 使用缓存的统计值。
- 异步统计。
- 分区/分表后分别统计再汇总。

## 排序字段设计

分页排序要稳定。只按 `created_at` 排序可能有重复时间，导致翻页重复或遗漏。

建议加唯一字段兜底：

```sql
ORDER BY created_at DESC, id DESC
```

游标也要带两个字段：

```text
cursor = last_created_at + last_id
```

## ES 深分页

ES 中 `from + size` 深分页也很重，因为每个分片都要取大量数据再合并。

优化：

- `search_after`。
- PIT point in time。
- scroll 用于离线遍历，不适合普通用户实时翻页。
- 限制最大翻页深度。

## 大批量任务的并发控制

旧资料中“大量数据优化分页”主要讲 `WaitGroup + channel` 控制 goroutine 并发。这个技巧适合处理批量数据任务：

```go
sem := make(chan struct{}, 10)
var wg sync.WaitGroup

for _, item := range items {
    item := item
    sem <- struct{}{}
    wg.Add(1)
    go func() {
        defer wg.Done()
        defer func() { <-sem }()
        handle(item)
    }()
}

wg.Wait()
```

注意：

- 控制并发数是为了保护数据库、下游接口、磁盘或网络。
- 不要无脑为每条数据启动 goroutine。
- 失败重试、超时和取消要设计清楚。

## 查询优化 checklist

- 是否只查询必要字段。
- where 条件是否命中索引。
- order by 是否能利用索引。
- 是否有大 offset。
- 是否有函数、隐式转换导致索引失效。
- 是否需要覆盖索引。
- 是否能用游标分页。
- 是否需要缓存或异步化。
- 是否是 OLTP 数据库不该承担的分析查询。

## 面试追问

### 为什么 offset 越大越慢？

数据库需要扫描并跳过 offset 指定的行，即使只返回 20 条，也可能已经读取和排序了大量数据。

### 游标分页缺点是什么？

不能方便随机跳页；排序字段必须稳定；客户端需要保存上一页游标；复杂筛选下索引设计要求更高。

### 如何设计订单列表分页？

按用户查询订单时，建立 `(user_id, created_at, id)` 联合索引，使用 `created_at + id` 做游标分页，避免深 offset。订单详情再按 ID 查询。
