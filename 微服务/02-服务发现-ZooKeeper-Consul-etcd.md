# 服务发现、ZooKeeper、Consul 与 etcd

## 分布式协调工具解决什么问题

ZooKeeper、Consul、etcd 都常用于分布式系统中的协调场景：

- 服务发现。
- 配置管理。
- 分布式锁。
- Leader 选举。
- 健康检查。
- 元数据存储。

它们不是普通缓存，也不是业务数据库，而是偏基础设施的协调组件。

## ZooKeeper

ZooKeeper 是经典分布式协调服务。

核心特点：

- 层级命名空间，类似文件系统。
- 节点称为 znode。
- 支持临时节点、顺序节点、watch 机制。
- 使用 ZAB 协议保证一致性。
- 在 Kafka、Hadoop 等生态中历史应用广泛。

适合：

- 分布式锁。
- Leader 选举。
- 配置中心。
- 集群元数据管理。

分布式锁常用做法：

1. 在锁路径下创建临时顺序节点。
2. 序号最小者获得锁。
3. 其他客户端 watch 自己前一个节点。
4. 持锁客户端断开会话后临时节点自动删除。

优点：公平性较好，锁释放可靠。

缺点：实现和运维复杂，watch 使用不当会有惊群问题。

## etcd

etcd 是强一致的分布式 KV 存储，Kubernetes 使用它保存集群状态。

核心特点：

- 基于 Raft。
- 强一致。
- 支持 lease 租约。
- 支持 watch。
- API 简洁。

适合：

- Kubernetes 元数据。
- 服务发现。
- 配置管理。
- 分布式锁。
- Leader 选举。

etcd 分布式锁通常基于：

- lease：客户端失联后锁自动过期。
- revision：根据创建顺序判断锁拥有者。
- watch：监听前一个 key 删除。

## Consul

Consul 更偏服务发现和服务治理。

核心特点：

- 服务注册与发现。
- 健康检查。
- KV 存储。
- 多数据中心支持。
- Gossip 协议做成员发现。
- Raft 保证核心状态一致性。
- 支持服务网格能力。

适合：

- 微服务注册发现。
- 健康检查。
- 多数据中心服务目录。
- 配置管理。

## 三者对比

| 对比项 | ZooKeeper | etcd | Consul |
| --- | --- | --- | --- |
| 一致性协议 | ZAB | Raft | Gossip + Raft |
| 数据模型 | 树形 znode | KV | 服务目录 + KV |
| 服务发现 | 可实现 | 可实现 | 原生强项 |
| 配置管理 | 支持 | 支持 | 支持 |
| 分布式锁 | 成熟 | 成熟 | 可实现 |
| 多数据中心 | 一般 | 一般 | 强 |
| 典型生态 | Hadoop/Kafka 旧生态 | Kubernetes | 微服务治理 |

选择建议：

- Kubernetes 相关：etcd。
- 传统分布式协调、已有生态：ZooKeeper。
- 服务发现、健康检查、多数据中心：Consul。

## 一致性

### ZooKeeper 强一致体现

- 单 leader 处理写请求。
- ZAB 保证事务顺序广播。
- 多数节点确认后提交。
- zxid 表示全局事务顺序。

### etcd 强一致体现

- Raft leader 处理写请求。
- 日志复制到多数节点后提交。
- lease 和 revision 保证可观察顺序。
- 线性一致读可确保读到最新提交结果。

### Consul 一致性

Consul 用 Gossip 管理成员状态传播，用 Raft 保证 server 节点上的一致性数据。服务健康信息传播与强一致 KV 读写要区分理解。

## 分布式锁对比：Redis、ZooKeeper、etcd

| 对比项 | Redis | ZooKeeper | etcd |
| --- | --- | --- | --- |
| 一致性 | 较弱，依赖部署和客户端实现 | 强 | 强 |
| 自动释放 | TTL | 临时节点 | Lease |
| 公平性 | 通常不保证 | 临时顺序节点可保证 | revision 顺序可实现 |
| 性能 | 高 | 中 | 中 |
| 复杂度 | 低 | 高 | 中 |
| 适用 | 短锁、弱一致 | 强协调 | 强协调、云原生 |

面试回答：Redis 锁简单高性能，但强一致性不如基于共识协议的 ZooKeeper/etcd。

## Leader 选举

Leader 选举用于多个实例中选出一个主节点执行任务。

常见场景：

- 定时任务只允许一个实例执行。
- 主从切换。
- 分片调度。
- 集群控制面。

实现方式：

- ZooKeeper 临时顺序节点。
- etcd lease + compare-and-swap。
- Consul session。

## 服务发现流程

典型服务发现：

1. 服务启动后注册自身地址和元信息。
2. 注册中心定期健康检查。
3. 消费方查询服务列表。
4. 客户端或网关做负载均衡。
5. 服务下线或健康检查失败后从列表移除。

注意：

- 服务发现要配合心跳/租约。
- 客户端要缓存服务列表，避免注册中心抖动影响全站。
- 调用方仍要设置超时、重试、熔断。

## 面试追问

### 为什么 etcd 常用于 Kubernetes？

因为 Kubernetes 需要一个强一致、高可用、可 watch 的 KV 存储保存集群状态，etcd 的 Raft、watch、lease、revision 模型非常契合。

### ZooKeeper watch 是永久的吗？

不是。ZooKeeper watch 通常是一次性触发，需要客户端重新注册。使用时要处理断线重连和事件丢失后的重新同步。

### 服务发现和负载均衡是什么关系？

服务发现负责知道有哪些实例可用，负载均衡负责从可用实例中选择一个进行调用。两者通常配合使用。
