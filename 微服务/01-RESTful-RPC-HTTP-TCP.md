# RESTful、RPC、HTTP 与 TCP

## TCP 三次握手

TCP 是面向连接、可靠、基于字节流的传输层协议。建立连接需要三次握手：

1. 客户端发送 SYN，表示希望建立连接，并发送初始序列号。
2. 服务端回复 SYN + ACK，表示收到客户端请求，并发送自己的初始序列号。
3. 客户端回复 ACK，连接建立。

为什么是三次？

- 双方都要确认自己的发送能力和接收能力正常。
- 双方都要同步初始序列号。
- 两次握手无法避免历史连接请求造成的误连接问题。

## TCP 四次挥手

断开连接通常需要四次挥手：

1. 主动关闭方发送 FIN，表示自己不再发送数据。
2. 被动关闭方回复 ACK，表示收到关闭请求。
3. 被动关闭方处理完剩余数据后发送 FIN。
4. 主动关闭方回复 ACK，进入 TIME_WAIT。

为什么通常是四次？

TCP 是全双工连接。主动方不发数据了，不代表被动方也立即没有数据要发，所以 ACK 和 FIN 常分开发送。

## TIME_WAIT

主动关闭方会进入 TIME_WAIT。

作用：

- 确保最后一个 ACK 能被对方收到。
- 让旧连接中的延迟报文在网络中自然消失，避免污染新连接。

面试追问：

- TIME_WAIT 过多可能消耗端口资源。
- 服务端大量 TIME_WAIT 可能说明连接关闭策略、短连接或负载均衡配置需要检查。

## HTTP/1.1 与 HTTP/2

| 对比项 | HTTP/1.1 | HTTP/2 |
| --- | --- | --- |
| 传输格式 | 文本 | 二进制帧 |
| 连接复用 | 支持 keep-alive，但请求响应队头阻塞明显 | 一个连接上多路复用多个 stream |
| 头部压缩 | 无内置高效压缩 | HPACK 头部压缩 |
| 服务端推送 | 不支持 | 支持，但实际使用有限 |
| 流优先级 | 不支持 | 支持优先级机制 |

纠错：不能说 HTTP/1.1 每个请求都必须新建 TCP 连接。HTTP/1.1 默认支持长连接，但同一连接上的请求响应仍容易产生队头阻塞。HTTP/2 解决了应用层多请求复用问题，但在 TCP 层丢包时仍可能受 TCP 队头阻塞影响。HTTP/3 基于 QUIC 进一步改善这一点。

## HTTP 常见状态码

- 200 OK：成功。
- 201 Created：资源创建成功。
- 204 No Content：成功但无响应体。
- 301/302：重定向。
- 400 Bad Request：请求参数错误。
- 401 Unauthorized：未认证。
- 403 Forbidden：无权限。
- 404 Not Found：资源不存在。
- 409 Conflict：资源冲突。
- 429 Too Many Requests：限流。
- 500 Internal Server Error：服务端错误。
- 502 Bad Gateway：网关收到无效响应。
- 503 Service Unavailable：服务不可用。
- 504 Gateway Timeout：网关超时。

## REST / RESTful

REST 是一种以资源为中心的架构风格，常基于 HTTP。

核心特点：

- 资源用 URI 表示。
- 使用 HTTP 方法表达动作。
- 无状态，每个请求携带足够上下文。
- 使用状态码表达结果。

示例：

```http
GET    /users/1
POST   /users
PUT    /users/1
PATCH  /users/1
DELETE /users/1
```

适合：

- 对外开放 API。
- 资源模型清晰的系统。
- 前后端通信。
- 简单、通用、可读性强的接口。

## RPC

RPC 关注“调用远程方法像调用本地方法”。

常见形式：

- gRPC。
- Thrift。
- JSON-RPC。
- Dubbo。

特点：

- 强接口定义。
- 客户端可生成调用代码。
- 性能通常更好，尤其是二进制协议。
- 适合服务内部高频调用。

gRPC 常使用 Protocol Buffers 定义接口：

```proto
service UserService {
  rpc GetUser(GetUserRequest) returns (GetUserResponse);
}
```

## REST 和 RPC 对比

| 对比项 | REST | RPC |
| --- | --- | --- |
| 抽象 | 资源 | 方法/服务 |
| 协议 | 通常 HTTP + JSON | HTTP/2、TCP、自定义协议等 |
| 可读性 | 强 | 取决于工具 |
| 性能 | JSON 开销较大 | 二进制协议通常更高效 |
| 适用范围 | 对外 API、资源操作 | 内部服务调用、高性能场景 |
| 演进 | 通过资源和字段兼容演进 | 依赖 IDL 和版本管理 |

选择建议：

- 外部开放 API：REST 更通用。
- 内部微服务高频调用：gRPC/RPC 更合适。
- 复杂查询或聚合接口：不必强行 REST，清晰可维护更重要。

## 幂等性

REST 面试常问幂等：

- GET：幂等，只读。
- PUT：幂等，完整替换。
- DELETE：幂等，多次删除结果一致。
- POST：通常非幂等，用于创建或提交动作。
- PATCH：不一定幂等，取决于实现。

实际业务要通过幂等键、唯一索引、状态机保证接口幂等。

## 超时、重试和熔断

网络调用必须考虑失败：

- 设置连接超时和读写超时。
- 重试只用于幂等或可安全重复的请求。
- 使用熔断保护下游。
- 使用限流保护自身。
- 传递 traceId 方便排查。

## 多实例 WebSocket 聊天服务

单实例 WebSocket 很简单：连接在哪台机器上，就由哪台机器往客户端写消息。多实例后，难点在于“用户连接可能在任意实例上”，A 实例收到消息时，接收方可能连接在 B 实例。

### 消息如何跨实例推送

常见轻量方案是 Redis Pub/Sub：

1. 客户端连接到某个 WebSocket 实例。
2. 实例在本机内存保存连接映射：`user_id -> conn/device/session`。
3. 用户发送聊天消息后，服务端先把消息落库，生成 `message_id`。
4. 服务端向 Redis channel 发布推送事件，例如 `{receiver_id, message_id, payload}`。
5. 所有 WebSocket 实例订阅该 channel，收到事件后检查接收方是否在本机在线。
6. 如果连接在本机，就通过 WebSocket 推给客户端。

一句话：Redis Pub/Sub 负责跨实例广播“有消息要推”，真正的连接仍由持有连接的 WebSocket 实例负责发送。

注意：Pub/Sub 不持久化消息，也没有 ACK。它适合做实时通知，不适合当可靠消息存储。

### 推送失败怎么处理

聊天消息不要只依赖 WebSocket 推送成功。

常见做法：

- 消息先落库，再发布推送事件。
- 客户端收到消息后回 ACK，服务端记录 `delivered/read` 状态或更新 `last_ack_message_id`。
- 推送失败、连接断开、客户端未 ACK 时，把消息保留为未读或待投递。
- 客户端重连后带上 `last_message_id` 或 `last_ack_message_id`，从服务端补拉缺失消息。
- 对可靠性要求更高时，用 Redis Stream、Kafka、RocketMQ 等替代单纯 Pub/Sub，提供消费确认、重试和死信处理。

面试里可以强调：WebSocket 推送只是实时通道，消息可靠性靠“消息存储 + ACK + 重连补拉 + 幂等”。

### 客户端状态如何保存

通常分三层：

| 状态 | 保存位置 | 说明 |
| --- | --- | --- |
| 当前连接 | WebSocket 实例本机内存 | `user_id -> conn`，最快，但实例重启会丢 |
| 在线状态 | Redis | `online:{user_id}`，设置 TTL，通过心跳续期 |
| 消息状态 | 数据库 | 消息内容、已送达、已读、未读数、最后确认位置 |

本机连接表不能放数据库，因为连接对象不能序列化，也不能被其他实例直接使用。Redis 在线状态也不要当强一致事实，只能作为“最近看起来在线”的参考。

### 常见多实例难点

- 负载均衡：WebSocket 是长连接，连接建立后会长期占用某个实例；需要关注连接数均衡和实例扩缩容。
- 用户多端登录：同一个用户可能有手机、网页、桌面端多个连接，推送要按设备或会话处理。
- 消息顺序：同一会话内通常需要递增序号或按消息 ID 排序，避免网络抖动导致乱序展示。
- 重复推送：Pub/Sub、重试、客户端重连都可能造成重复，客户端和服务端都要按 `message_id` 幂等去重。
- 大群 fan-out：一个群消息推给很多人时，不能让所有实例无脑全量广播，要考虑房间分片、批量拉取、异步 fan-out 或 MQ 削峰。
- 慢客户端背压：客户端网络慢时，服务端写缓冲会堆积；要限制单连接队列长度，必要时断开连接让客户端重连补拉。
- 心跳和断线检测：通过 ping/pong 或应用层心跳清理死连接，同时续期 Redis 在线状态。
- 灰度发布和重启：实例下线前要停止接收新连接，通知旧连接重连，避免直接断开造成大量失败。

简化回答：多实例聊天的核心不是“怎么拿到 socket”，而是“连接路由、消息可靠、状态同步、重连补偿、幂等去重”。

## Web API 鉴权和安全常见坑

### HTTPS / TLS

HTTPS = HTTP over TLS，解决传输加密、身份认证和完整性校验。它不能解决业务鉴权、越权、SQL 注入等应用层问题。

### Cookie、Session 和 JWT

- Session：服务端保存登录状态，客户端一般只保存 session id。
- JWT：令牌自包含用户信息和签名，服务端可无状态校验，但撤销和过期控制更麻烦。
- Cookie 适合浏览器自动携带，但要注意 `HttpOnly`、`Secure`、`SameSite`。
- Token 放在 localStorage 容易受 XSS 影响；放 Cookie 要注意 CSRF。

### CORS

CORS 是浏览器安全策略，不是服务端接口权限控制。服务端即使配置了 CORS，也仍要做鉴权和权限校验。

易错点：

- 401 是未认证，403 是已认证但无权限。
- HTTPS 只保护传输链路，不代表接口本身安全。
- 前端隐藏按钮不等于有权限，后端必须校验。

## 面试追问

### TCP 和 UDP 的区别？

TCP 面向连接、可靠、有序、基于字节流，适合文件传输、HTTP 等。UDP 无连接、不保证可靠和顺序、报文边界清晰，适合实时音视频、DNS、游戏等对延迟敏感场景。

### gRPC 为什么常用于微服务内部？

它有强类型 IDL、代码生成、HTTP/2 多路复用、二进制序列化、流式调用能力，适合内部服务间高性能通信。

### HTTP 2.0 是否完全解决队头阻塞？

解决了 HTTP/1.1 应用层请求队头阻塞，但由于仍基于 TCP，丢包时 TCP 层仍可能阻塞后续数据。HTTP/3 基于 QUIC 进一步改善。
