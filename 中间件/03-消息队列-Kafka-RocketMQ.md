# 消息队列、Kafka 与 RocketMQ

## 队列和消息中间件的区别

以 Laravel 内置队列和独立消息中间件为例：

- 队列更偏向后台任务处理，例如发邮件、图片处理、异步导出。
- 消息中间件更偏向系统间异步通信、解耦、削峰和事件驱动。

Laravel 队列可以使用 database、Redis、SQS 等驱动；Kafka、RocketMQ、RabbitMQ 则是独立消息系统，适合跨服务通信和更复杂的消息语义。

## 消息队列的核心作用

### 异步处理

把耗时但不需要同步完成的任务放到队列中，提高接口响应速度。

### 应用解耦

订单系统创建订单后只发送事件，不直接调用库存、积分、短信等多个系统。

```text
订单创建 -> MQ -> 库存服务
              -> 积分服务
              -> 通知服务
```

### 削峰填谷

高峰请求先进入 MQ，消费者按自身能力逐步处理，保护数据库和下游服务。

### 广播通知

一个事件被多个消费者订阅，各自处理不同逻辑。

## 引入 MQ 的代价

- 系统链路变长。
- 数据一致性从强一致变为最终一致。
- 需要处理重复消息、消息丢失、消息堆积。
- 运维复杂度增加。
- 排查问题需要 traceId 和全链路日志。

## 消息可靠性

完整链路包括：

1. 生产者发送成功。
2. Broker 持久化成功。
3. 消费者消费成功并提交确认。

常见手段：

- 生产端发送确认。
- Broker 多副本。
- 消费端手动 ack/commit。
- 失败重试。
- 死信队列。
- 定时补偿和对账。

## 消息重复和幂等

MQ 通常只能尽量保证“至少一次”，消费者要做好幂等。

幂等方案：

- 业务唯一键。
- 去重表。
- Redis set 记录消息 ID。
- 数据库唯一索引。
- 状态机流转校验。

示例：

```sql
INSERT INTO processed_message(message_id, created_at)
VALUES (?, NOW());
```

如果唯一键冲突，说明消息已处理过。

## 消息顺序

顺序消息通常要求同一业务 key 路由到同一个分区/队列。

例如同一订单的事件：

```text
order_created -> order_paid -> order_shipped
```

都按 `order_id` 路由到同一队列。

注意：全局顺序通常需要单分区/单队列，会牺牲吞吐。多数业务只需要局部顺序。

## 消息堆积

原因：

- 消费者处理慢。
- 下游数据库慢。
- 消费者异常。
- 流量突增。
- 分区/队列数量不足。

处理：

- 扩容消费者。
- 增加分区/队列。
- 优化消费逻辑和批处理。
- 临时降级非核心逻辑。
- 将积压消息转储后离线处理。

## Kafka

Kafka 是高吞吐分布式日志系统，适合大数据流、日志、事件流。

核心概念：

- Topic：主题。
- Partition：分区，提供并行和局部有序。
- Producer：生产者。
- Consumer Group：消费者组。
- Offset：消费位点。
- Broker：服务节点。

优势：

- 高吞吐。
- 分区扩展能力强。
- 日志存储模型简单可靠。
- 与 Flink、Spark、ClickHouse 等生态结合好。

不足：

- 运维和参数调优复杂。
- 延迟消息、复杂事务消息不是传统强项。
- 顺序性通常是分区内有序。

### Kafka offset 和 rebalance 踩坑

- offset 表示消费进度，提交 offset 不等于业务处理成功。
- 消费者应在业务处理成功后再提交 offset，否则可能丢消息。
- 如果业务处理成功但 offset 提交失败，消息可能重复消费，所以消费者必须幂等。
- Consumer Group 发生 rebalance 时，分区会重新分配，容易出现短暂暂停和重复消费。
- 分区数决定并行度上限，同一个分区同一时刻只能被同一消费组内一个消费者消费。

## RocketMQ

RocketMQ 是分布式消息中间件，常见于电商、金融、订单等业务系统。

核心概念：

- Producer。
- Consumer。
- Topic。
- MessageQueue。
- NameServer。
- Broker。

优势：

- 业务消息能力丰富。
- 支持事务消息。
- 支持延迟消息。
- 支持顺序消息。
- 消息过滤能力较强。

注意：RocketMQ 的顺序通常也是队列内顺序。所谓全局顺序需要单队列，会影响吞吐，不能简单说“天然支持无限制全局顺序”。

## RocketMQ 死信队列

RocketMQ 消费失败后会按照重试策略重新投递。超过最大重试次数仍然失败时，消息会进入该消费组对应的死信队列，通常也叫 DLQ。

常见特点：

- 死信队列按消费组维度生成，不是所有消费者共用一个队列。
- 进入死信队列说明消息多次消费失败，需要人工排查、修复数据或走补偿流程。
- 死信消息可以被单独查询、导出或重新投递，具体处理方式取决于业务和运维工具。
- 消费者逻辑仍然要保证幂等，避免死信重投后造成重复扣款、重复发货等问题。

适合处理：

- 下游服务持续异常导致消费失败。
- 数据格式异常、业务状态不合法。
- 依赖超时、接口限流、数据库异常等临时或半永久问题。

面试回答：

> 死信队列是 MQ 的兜底机制。消息消费多次失败后不会无限重试，而是进入死信队列，方便后续人工排查、修复和补偿。它不能替代业务幂等和补偿机制，只是把异常消息隔离出来，避免阻塞正常消费。

## Kafka 和 RocketMQ 对比

| 对比项 | Kafka | RocketMQ |
| --- | --- | --- |
| 定位 | 高吞吐日志/事件流平台 | 业务消息中间件 |
| 吞吐 | 很强 | 强 |
| 顺序 | 分区内有序 | 队列内有序 |
| 事务消息 | 支持事务和 EOS 语义，但使用复杂 | 事务消息是重要特性，业务集成更直观 |
| 延迟消息 | 原生能力相对弱，常需额外方案 | 支持延迟消息 |
| 大数据生态 | 非常成熟 | 相对弱 |
| 业务消息特性 | 相对克制 | 更丰富 |
| 典型场景 | 日志、埋点、流处理 | 订单、支付、交易、异步业务 |

## RocketMQ 应用解耦示例

不使用 MQ：

```text
订单服务 -> 库存服务 -> 积分服务 -> 短信服务
```

问题：

- 任一下游慢都会拖慢订单接口。
- 下游故障可能影响主流程。
- 新增订阅方需要改订单服务。

使用 MQ：

```text
订单服务 -> 发送 OrderCreated 事件 -> RocketMQ
库存服务消费事件扣库存
积分服务消费事件加积分
通知服务消费事件发短信
```

订单服务只关心事件发出，下游独立扩展。

## 事务消息

RocketMQ 事务消息常用于“本地事务 + 消息发送”一致性，解决的是业务数据已经提交，但消息没有成功发出去，或者消息发出后本地事务失败的问题。

大致流程：

1. 发送半消息，此时消费者不可见。
2. 执行本地事务，例如创建订单、扣减账户余额、写入业务表。
3. 本地事务成功则提交消息，失败则回滚消息。
4. Broker 对未知状态消息进行回查，生产者根据本地事务状态决定提交还是回滚。

典型使用场景：

- 订单创建成功后，通知库存、积分、优惠券、通知等下游系统。
- 支付成功落库后，发送支付成功事件给订单、会员、账务系统。
- 本地业务状态变更后，需要可靠触发异步流程，例如审核通过后发放权益。

使用边界：

- 事务消息保证的是本地事务和消息发送的一致性，不保证消费者一定处理成功。
- 消费者仍然要幂等，失败后依赖重试、死信队列、补偿任务和对账兜底。
- 不适合替代所有分布式事务，如果多个系统必须强一致提交，仍要考虑 TCC、Saga、事务表、可靠消息最终一致性或强一致协调方案。

## 选型建议

选择 Kafka：

- 海量日志和埋点。
- 流处理。
- 与大数据生态集成。
- 极高吞吐优先。

选择 RocketMQ：

- 电商订单。
- 事务消息。
- 延迟任务。
- 业务消息语义丰富。

选择 Redis Stream/List：

- 小规模、轻量队列。
- 已有 Redis 且可靠性要求不高或可接受补偿。

选择 RabbitMQ：

- 复杂路由。
- AMQP 生态。
- 中小规模业务消息。

## 面试追问

### 如何保证消息不丢？

生产者确认、Broker 持久化和副本、消费者手动 ack、失败重试、死信队列、补偿扫描。业务上还要通过对账兜底。

### 如何保证消息不重复消费？

严格不重复很难，工程上通常保证消费者幂等。用消息 ID、业务唯一键、去重表或状态机防重复。

### MQ 如何削峰？

请求先写入 MQ，消费者按固定能力处理，数据库承受的是平滑后的消费速度。必要时结合限流、降级和库存预扣。
