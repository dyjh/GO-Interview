# Redis 分布式锁、限流与一致性

## Redis 分布式锁

Redis 分布式锁的核心目标是：多个进程、多个服务实例甚至多台机器竞争同一份资源时，只允许一个执行关键逻辑。

常见场景：防重复提交、定时任务抢占、短时间互斥、库存预扣前置保护等。

### SETNX 的问题

旧写法：

```text
SETNX lock_key value
EXPIRE lock_key 10
```

问题是 `SETNX` 成功后，如果客户端在执行 `EXPIRE` 前宕机，锁就没有过期时间，可能永久死锁。

正确基础写法：

```text
SET lock_key unique_value NX EX 10
```

这会把“只在不存在时设置”和“设置过期时间”合成一个原子命令。

### 安全释放锁

释放锁不能简单 `DEL lock_key`，否则可能误删别人的锁。

典型问题：

1. 客户端 A 获取锁，业务执行超时，锁自动过期。
2. 客户端 B 获取同一把锁。
3. 客户端 A 执行结束后直接 `DEL`，误删 B 的锁。

正确做法是设置唯一值，释放时校验 value。

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

Lua 可以保证“判断 value + 删除锁”整体原子执行。

### 锁超时和续期

锁 TTL 太短，业务没执行完锁就释放；TTL 太长，故障后恢复慢。

常见处理：

- 估算业务最大耗时，设置合理 TTL。
- 使用看门狗续期机制。
- 业务操作设计成幂等。
- 关键一致性场景优先考虑数据库锁、etcd、ZooKeeper 等更强一致的方案。

### 适用场景

适合：

- 短时间互斥。
- 允许失败重试。
- 对强一致要求不是极端严格。
- 锁竞争不太激烈。

谨慎使用：

- 金融扣款。
- 强一致库存。
- 长事务。
- 网络分区下必须严格互斥的场景。

### Redlock 简述

Redlock 是 Redis 官方提出的多节点分布式锁算法：在多个独立 Redis 节点上获取锁，并要求多数节点成功。

面试回答建议：

- Redlock 试图提高单 Redis 锁在故障场景下的可靠性。
- 它仍有争议，特别是在时钟漂移、网络分区、进程暂停等场景。
- 对极强一致要求的锁，etcd、ZooKeeper 这类基于共识协议的系统通常更稳妥。

## Redis 限流

Redis 限流常用于控制请求频率，保护接口、数据库、第三方服务或下游系统。

常见接入位置：

- API 网关层：适合微服务统一入口、租户/API Key/IP 维度限流。
- 应用路由中间件：适合普通 API 服务按用户、接口、业务规则做细粒度限流。
- Nginx/OpenResty：适合更前置地挡住恶意流量，通常需要 Lua 接 Redis。
- 服务内部：适合保护某个下游资源，例如第三方 API、数据库写入、MQ 消费速度。

常见限流算法：固定窗口、滑动窗口、令牌桶、漏桶。

### 固定窗口计数器

按固定时间窗口计数，例如每分钟最多 60 次：

```text
INCR rate:user:1:202606232200
EXPIRE rate:user:1:202606232200 60
```

优点：简单，成本低。

缺点：窗口边界可能突刺。例如 59 秒末和 60 秒初各打满一次，短时间内流量翻倍。

### 滑动窗口

用 sorted set 保存请求时间戳：

```text
ZADD rate:user:1 now request_id
ZREMRANGEBYSCORE rate:user:1 0 now-window
ZCARD rate:user:1
EXPIRE rate:user:1 window
```

需要 Lua 保证原子性。

优点：比固定窗口更平滑。

缺点：内存和计算成本高于固定窗口。

### 令牌桶

令牌按固定速率生成，请求需要先拿到令牌才能通过。

它解决的问题是：**限制平均速率，同时允许短时间突发**。

适合：

- API 接口限流。
- 用户、商户、租户维度限流。
- 允许短时间突发，但整体 QPS 要受控的场景。

Redis 实现时通常用 Lua 维护：

- 当前令牌数。
- 上次填充时间。
- 桶容量。
- 令牌生成速率。

### 漏桶

请求进入桶，按固定速率流出。

它解决的问题是：**削平突发流量，让下游按稳定速度处理请求**。

适合：

- 调第三方接口，有固定 QPS 限制。
- 控制 MQ 消费速度。
- 写数据库前削峰。
- 短信、邮件等供应商限速场景。

### 算法选择

| 需求 | 推荐算法 |
| --- | --- |
| 简单计数，能接受边界突刺 | 固定窗口 |
| 想让窗口边界更平滑 | 滑动窗口 |
| 限制平均速率，同时允许短暂突发 | 令牌桶 |
| 希望请求按固定速度进入下游 | 漏桶 |

一句话记：令牌桶是“有通行证才能过，证可以攒一点”；漏桶是“请求排队，按固定速度放出去”。

## Redis WATCH

`WATCH` 提供乐观锁机制：

```text
WATCH key
GET key
MULTI
SET key new_value
EXEC
```

如果 WATCH 的 key 在 `EXEC` 前被其他客户端修改，事务执行失败。

注意：

- Redis 事务不支持传统数据库那种自动回滚。
- `MULTI/EXEC` 保证命令顺序执行，但命令本身错误需要业务处理。
- 高竞争场景下 WATCH 失败率高，可能需要重试或换方案。

## 高并发下数据一致性

Redis 高并发一致性常见手段：

- 单个命令天然原子。
- 多命令组合用 Lua。
- 乐观锁 WATCH。
- 分布式锁。
- 队列串行化。
- 数据库最终兜底。

库存扣减示例：

```lua
local stock = tonumber(redis.call("GET", KEYS[1]))
if stock == nil or stock <= 0 then
    return 0
end
redis.call("DECR", KEYS[1])
return 1
```

Redis 预扣成功后，再通过 MQ 异步创建订单和落库；失败时要补偿库存。

### 拼团/秒杀锁优化

旧资料中提到拼团高并发下 Redis `setnx` 可能有问题。优化方向：

- 不要所有请求竞争同一把锁，按活动 ID、商品 ID、用户 ID 分片。
- 热点库存用 Redis 原子扣减，减少数据库锁。
- 使用 Lua 合并检查和扣减。
- 请求先进队列削峰。
- 下单接口做幂等，避免重复下单。
- 锁等待设置超时，失败快速返回。
- 监控锁等待时间、失败率、超时率。

### Redis 与数据库一致性

Redis 常作为缓存或预处理层，最终真实状态一般落到数据库。

推荐设计：

1. Redis 做快速校验和预扣。
2. MQ 异步化下单。
3. 数据库事务写订单和扣库存。
4. 失败消息进入重试或补偿。
5. 定时任务对账，修正 Redis 和 DB。

## 易错点总结

- 单个 `SETNX` 不等于完整分布式锁，还要处理过期时间、唯一标识、误删和续期。
- Redis 锁不一定绝对安全，强一致场景要谨慎。
- Lua 原子性只保证 Redis 内部脚本执行不被插入，不代表跨系统事务一致。
- 限流越靠前越能保护后端，越靠近业务越能做精细规则。
- 秒杀类场景不要让所有请求竞争同一把大锁，应尽量用原子扣减、分片、队列削峰和幂等兜底。

## 面试追问

### 分布式锁和单机使用 SETNX 有什么区别？单机 SETNX 能叫分布式锁吗？

关键看锁服务的使用场景，而不是只看 Redis 是单机还是集群。

- 如果只有一个进程、一个实例在本机内部做互斥，`SETNX` 只能算单机锁或本地互斥方案。
- 如果多个服务实例、多进程、多机器通过同一个 Redis key 竞争同一把锁，即使 Redis 本身是单节点，也可以算一种 Redis 分布式锁实现。
- 但单 Redis 节点可靠性有限，存在单点故障、主从切换丢锁、网络分区、客户端长时间暂停等问题。

面试可以这样答：

> 单机 SETNX 本身只是 Redis 的原子命令，不等于完整的分布式锁。多个分布式节点基于 Redis 的同一个 key 做互斥时，才是在实现分布式锁。单 Redis 也能实现基础分布式锁，但可靠性不如基于共识协议的 etcd/ZooKeeper，也要处理过期时间、唯一标识、误删和续期问题。

### Redis 锁一定安全吗？

不是。单 Redis 锁在主从切换、网络分区、长 GC、客户端暂停等场景可能失效。要根据业务一致性要求选择方案。强一致锁优先考虑 etcd、ZooKeeper 或数据库事务。

### Lua 为什么能保证原子性？

Redis 执行 Lua 脚本时会把脚本作为一个整体执行，中间不会插入其他命令，因此适合把多个 Redis 操作组合成一个原子操作。

### 限流放在哪里？

可放在客户端、Nginx/API 网关、应用服务、Redis 或专门限流服务。越靠前越能保护后端资源；越靠近业务越能做精细规则。微服务里常放网关层，普通 API 常放路由中间件。
