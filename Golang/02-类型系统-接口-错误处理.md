# 类型系统、接口与错误处理

## 类型系统核心思想

Go 的类型系统强调简单、显式和组合：

- 没有传统类继承，使用 struct 组合和接口抽象。
- 接口是隐式实现的，只要方法集匹配就实现了接口。
- 方法可以定义在自定义类型上。
- 类型转换必须显式进行。

这让 Go 代码通常更偏向“小接口 + 组合 + 明确依赖”。

## 接口 interface

接口定义一组方法签名：

```go
type Reader interface {
    Read(p []byte) (n int, err error)
}
```

任何类型只要实现 `Read` 方法，就隐式实现了 `Reader`。

```go
type FileReader struct{}

func (FileReader) Read(p []byte) (int, error) {
    return 0, io.EOF
}
```

面试回答重点：

- Go 接口是行为抽象，而不是父类。
- 实现接口不需要显式声明。
- 接口适合定义在使用方，而不是实现方。
- 小接口更符合 Go 风格，例如 `io.Reader`、`io.Writer`。

## 接口底层结构

面试中可以这样简化描述：

接口值由两部分组成：

- 动态类型。
- 动态值。

只有当动态类型和动态值都为空时，接口才等于 nil。

经典坑：

```go
type MyError struct{}

func (e *MyError) Error() string { return "my error" }

func f() error {
    var e *MyError = nil
    return e
}

func main() {
    err := f()
    fmt.Println(err == nil) // false
}
```

原因是返回的接口 `error` 中已经有了动态类型 `*MyError`，只是动态值为 nil。

## 空接口和 any

`interface{}` 可以表示任意类型。`any` 是 `interface{}` 的别名：

```go
func Print(v any) {
    fmt.Println(v)
}
```

空接口适合通用容器、日志字段、JSON 解析等场景，但业务代码中过度使用会丢失类型安全。

## 类型断言和 type switch

类型断言：

```go
v, ok := x.(string)
if !ok {
    return errors.New("not string")
}
```

类型分支：

```go
switch v := x.(type) {
case string:
    fmt.Println("string", v)
case int:
    fmt.Println("int", v)
default:
    fmt.Println("unknown")
}
```

注意：直接 `v := x.(string)` 在类型不匹配时会 panic，面试中建议使用 `comma ok` 形式。

## 方法集

方法集决定某个类型是否实现接口。

简化规则：

- 类型 `T` 的方法集包含接收者为 `T` 的方法。
- 类型 `*T` 的方法集包含接收者为 `T` 和 `*T` 的方法。

因此，如果接口方法由指针接收者实现，只有 `*T` 实现该接口，`T` 不一定实现。

```go
type Closer interface {
    Close() error
}

type File struct{}

func (f *File) Close() error { return nil }

var _ Closer = (*File)(nil) // OK
// var _ Closer = File{}    // 编译失败
```

## 错误处理

Go 推荐用显式 `error` 返回错误：

```go
data, err := os.ReadFile("config.yaml")
if err != nil {
    return fmt.Errorf("read config: %w", err)
}
```

设计原则：

- 可预期错误用 `error`。
- 不要用 panic 表达普通业务错误。
- 错误要带上下文，方便定位。
- 包装错误时用 `%w`，方便上层用 `errors.Is` 和 `errors.As` 判断。

## errors.Is / errors.As

```go
if errors.Is(err, os.ErrNotExist) {
    // 文件不存在
}
```

```go
var pathErr *os.PathError
if errors.As(err, &pathErr) {
    fmt.Println(pathErr.Path)
}
```

区别：

- `errors.Is` 判断错误链中是否包含某个哨兵错误。
- `errors.As` 从错误链中提取某种具体错误类型。

## panic 和 recover

`panic` 表示程序进入异常流程，当前 goroutine 停止正常执行，开始执行 defer 链。`recover` 只能在 defer 中生效。

```go
func safeRun(fn func()) (err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("panic: %v", r)
        }
    }()
    fn()
    return nil
}
```

使用建议：

- 框架边界、goroutine 顶层可以 recover，防止整个服务被单个请求打崩。
- 业务分支错误应返回 error。
- recover 后要记录堆栈，否则排障困难。

## 指针和值

Go 参数传递默认是值传递。传指针可以避免复制，并允许修改原对象。

```go
func rename(u *User, name string) {
    u.Name = name
}
```

注意点：

- slice/map/channel 本身是描述符值，复制描述符后仍指向共享底层数据。
- struct 值传递会复制整个结构体，字段较多时可能有成本。
- 指针不是越多越好，逃逸到堆上会增加 GC 压力。

## 泛型简述

Go 泛型适合容器、算法、通用工具函数：

```go
func Max[T ~int | ~int64 | ~float64](a, b T) T {
    if a > b {
        return a
    }
    return b
}
```

面试中可以强调：

- 泛型解决重复代码和类型安全问题。
- 不应该为了“高级”而过度泛型化。
- 业务代码中接口仍然常用于行为抽象，泛型常用于类型参数化。

## 易错点总结

- 接口 nil 判断要看动态类型和动态值。
- 空接口不是泛型的完全替代，过度使用会丢类型信息。
- 指针接收者会影响接口实现的方法集。
- panic 不是普通错误处理机制。
- `fmt.Errorf("xxx: %w", err)` 才能保留错误链。
- 值传递不等于底层数据一定不共享，slice/map/channel 要特别注意。
