# Golang 场景应用问题整合

这篇用于收集 Go 面试和项目中常见的应用场景类问题，重点关注“怎么设计”和“怎么落地”。

## 队列任务如何支持用户主动中断

接入 RocketMQ、Kafka、RabbitMQ 这类队列后，如果前端任务列表要支持“中断运行中的任务”，不要理解成从外部强杀 goroutine。Go 没有安全的 goroutine kill 机制，正确方案是：**任务状态机 + 分布式取消信号 + context 协作式退出**。

### 队列场景的核心流程

1. 创建任务时先写任务表，例如 `AutoJob`，状态为 `pending`，再投递 MQ 消息，消息里只放 `job_id`。
2. 消费者收到消息后，通过数据库乐观更新把任务从 `pending` 改为 `running`，避免重复消费同时跑同一个任务。
3. 消费者为该任务创建 `context.WithCancel`，并在本 worker 内记录 `job_id -> cancelFunc`。
4. 前端点击中断时，API 把任务状态改为 `canceling`，并发送取消消息或 Redis pub/sub 事件。
5. 各 worker 收到取消事件后，如果本机正在执行该 `job_id`，就调用对应的 `cancelFunc`。
6. 任务代码在步骤边界和阻塞 I/O 中感知 `ctx.Done()`，退出后把任务状态改为 `canceled`。

状态可以简化为：

```text
pending -> running -> success
pending -> running -> canceling -> canceled
pending -> running -> failed
```

取消不是失败。任务主动取消后通常应该 ack MQ 消息，返回消费成功，避免 MQ 把已取消任务重新投递。

### 不要到处手写 ctx 检查

任务确实要检查取消，但不需要每一步都散落一段 `select`。更好的方式是把耗时操作抽象成 step，统一在执行器里检查：

```go
type Step struct {
    Name string
    Run  func(ctx context.Context) error
}

func RunSteps(ctx context.Context, steps []Step) error {
    for _, step := range steps {
        if err := ctx.Err(); err != nil {
            return err
        }

        updateProcess(step.Name)

        if err := step.Run(ctx); err != nil {
            return err
        }

        if err := ctx.Err(); err != nil {
            return err
        }
    }
    return nil
}
```

同步镜像任务可以组织成：

```go
steps := []Step{}

for _, rack := range racks {
    for _, image := range images {
        rack := rack
        image := image
        steps = append(steps, Step{
            Name: "同步镜像",
            Run: func(ctx context.Context) error {
                return app.SyncImageWithContext(ctx, rack, image)
            },
        })
    }

    rack := rack
    steps = append(steps, Step{
        Name: "重启被控容器",
        Run: func(ctx context.Context) error {
            return restartRackControllerContainer(ctx, rack)
        },
    })
}

err := RunSteps(ctx, steps)
```

### 阻塞操作必须支持 context

如果 `SyncImage` 内部是 HTTP 请求，要改成 `SyncImageWithContext`，使用 `http.NewRequestWithContext` 或 `req.WithContext(ctx)`。数据库操作使用 `QueryContext/ExecContext`，外部命令使用 `exec.CommandContext`。

如果某个 SDK 或远端接口不支持 context，用户点中断后不能立即打断当前阻塞调用，只能等这次调用返回后，在下一个 step 边界退出。因此长任务最好拆成多个小步骤，每步之间都能感知取消。

### 兜底机制

队列任务还要考虑 worker 宕机和重复消费：

- 任务运行中定期更新心跳和进度。
- 定时扫描长时间无心跳的 `running/canceling` 任务，标记失败或重新投递。
- 每个 step 要尽量幂等，已经产生副作用的操作要有补偿逻辑。
- 消费者启动任务前用数据库状态流转抢占任务，避免同一个 `job_id` 被多个 worker 同时执行。

面试回答可以浓缩成一句：用户中断队列任务，本质是把任务改为 `canceling`，通知持有任务的 worker 调用 `cancelFunc`，任务通过 `context` 在 step 边界和阻塞 I/O 中协作式退出，最后把状态落为 `canceled`。

## GORM Save 和零值字段更新

这个问题要区分 `Save` 和 `Updates`。

`Save` 在主键存在时会更新所有字段，包括零值字段：

```go
user := User{
    ID:     1,
    Name:   "",
    Age:    0,
    Active: false,
}

db.Save(&user)
```

上面这种写法中，`Name=""`、`Age=0`、`Active=false` 都会参与更新。`Save` 更像“保存整个对象”：没有主键时创建，有主键时全字段更新。

真正容易忽略零值的是 `Updates(struct)`：

```go
db.Model(&User{}).
    Where("id = ?", 1).
    Updates(User{Name: "", Age: 0, Active: false})
```

这种写法默认只更新非零值字段，所以空字符串、`0`、`false` 可能不会更新。

如果确实要更新零值，可以用 `map` 或 `Select`：

```go
db.Model(&User{}).
    Where("id = ?", 1).
    Updates(map[string]any{
        "name":   "",
        "age":    0,
        "active": false,
    })
```

```go
db.Model(&User{}).
    Where("id = ?", 1).
    Select("name", "age", "active").
    Updates(User{Name: "", Age: 0, Active: false})
```

面试回答可以这样说：`Save` 会保存所有字段，零值也会更新；`Updates` 如果传 struct，默认跳过零值；如果要更新零值，使用 `map`、`Select` 或单字段 `Update`。
