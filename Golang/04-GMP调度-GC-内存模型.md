# GMP 调度、GC 与内存模型

## GMP 模型

Go runtime 使用 GMP 模型调度 goroutine：

- G：goroutine，表示待执行的任务。
- M：machine，对应操作系统线程。
- P：processor，逻辑处理器，持有本地运行队列和调度资源。

`GOMAXPROCS` 决定同时参与执行 Go 代码的 P 的数量。M 必须绑定 P 才能执行 Go 代码。

## 调度流程

简化流程：

1. 新建 goroutine 时，G 通常放入当前 P 的本地队列。
2. M 绑定 P，从 P 的本地队列取 G 执行。
3. 本地队列为空时，会尝试从全局队列或其他 P 的队列偷取任务。
4. G 发生网络 I/O、channel 阻塞、锁等待、系统调用等情况时，调度器切换到其他可运行 G。
5. 长时间运行的 G 会被抢占，避免一直占用 P。

## 工作窃取

工作窃取用于负载均衡：

- 如果某个 P 的本地队列为空，会从其他 P 的本地队列偷取一部分 G。
- 这样可以减少全局队列争用，也能让多个 P 尽量都有活干。

## 系统调用时发生什么

当 G 执行阻塞系统调用时：

- M 可能会进入 syscall 阻塞。
- P 会和这个 M 分离，交给其他 M 继续执行队列中的 G。
- 系统调用返回后，原来的 G 会尝试重新获得 P 继续执行。

这也是 Go 能在大量阻塞 I/O 场景下保持高并发的重要原因。

## 协作式调度和抢占式调度

Go 的调度既包含协作式调度，也包含 runtime 的抢占式调度。

协作式调度指 goroutine 在运行到某些调度点时主动或间接让出执行权，例如：

- channel 发送、接收阻塞。
- mutex、select、网络 I/O 等等待。
- 函数调用、栈增长检查等 runtime 可以插入调度检查的位置。

抢占式调度指 goroutine 正在执行 Go 代码且长时间没有主动让出 CPU 时，runtime 可以主动打断它，让当前 P 去运行其他可执行的 G。这样可以避免某个 CPU 密集型 goroutine 一直占着 P，导致其他 goroutine 长时间得不到执行机会。

抢占的大致过程：

1. runtime 的监控线程或调度器发现某个 G 运行时间过长。
2. runtime 标记这个 G 需要被抢占，必要时通过异步抢占机制打断正在运行的 M。
3. 被抢占的 G 在安全位置保存现场，重新放回可运行队列。
4. 当前 P 可以继续调度其他 G 执行。

需要注意，阻塞系统调用时 P 和 M 分离不是典型意义上的抢占式调度。系统调用场景下，是 G/M 已经进入内核阻塞，runtime 把 P 释放出来给其他 M 使用；抢占式调度则是 G 仍在执行 Go 代码时，被 runtime 主动暂停。

面试回答时可以说：

- Go 调度不是简单的线程池。
- 它既有协作式让出，也有 runtime 的抢占机制。
- 抢占提升了调度公平性，避免单个 goroutine 长时间占用 CPU。
- 阻塞 syscall 释放 P 和抢占式调度都能提升并发能力，但二者不是同一个概念。

## goroutine 和线程的区别

| 对比项 | goroutine | OS 线程 |
| --- | --- | --- |
| 创建成本 | 很低 | 较高 |
| 栈 | 初始小，可增长 | 通常较大 |
| 调度者 | Go runtime | 操作系统 |
| 数量 | 可轻松创建大量 | 数量受系统资源限制更明显 |
| 通信方式 | channel、共享内存 | 共享内存、系统 IPC |

## 进程、线程、协程

| 特性 | 进程 | 线程 | 协程/goroutine |
| --- | --- | --- | --- |
| 资源隔离 | 强 | 同进程内共享 | 同进程内共享 |
| 创建/切换成本 | 高 | 中 | 低 |
| 调度层级 | OS | OS | 用户态 runtime 为主 |
| 通信 | IPC 成本较高 | 共享内存，需要同步 | channel 或共享内存 |
| 适用场景 | 强隔离、多服务 | CPU 并行、共享状态 | 高并发 I/O、轻量任务 |

## GC 垃圾回收

Go 使用并发标记清扫 GC。核心目标是在自动内存管理的同时控制暂停时间。

简化阶段：

1. 标记准备：短暂停顿，开启写屏障。
2. 并发标记：GC 和用户 goroutine 并发执行，找出可达对象。
3. 标记终止：短暂停顿，完成标记。
4. 清扫：回收不可达对象占用的内存。

面试关键词：

- 三色标记。
- 写屏障。
- 并发标记。
- STW 时间控制。
- GC pacer 根据堆增长目标调节回收节奏。

## GC 触发时机

常见触发方式：

- 堆内存增长达到 GC 目标。
- 手动调用 `runtime.GC()`。
- runtime 根据内存压力和调度情况触发。

`GOGC` 控制 GC 目标增长比例。默认值通常表示堆在上次 GC 后增长一定比例再触发下一轮。调大 `GOGC` 可减少 GC 频率但增加内存占用，调小则相反。

## 逃逸分析

逃逸分析决定变量分配在栈上还是堆上：

- 如果对象生命周期只在函数内，通常可分配在栈上。
- 如果对象被返回、被闭包捕获、存入接口、或编译器无法证明生命周期，可能逃逸到堆上。

查看逃逸：

```bash
go build -gcflags="-m" ./...
```

面试回答：

逃逸到堆上不是错误，但会增加 GC 压力。性能敏感代码需要减少不必要分配，例如复用 buffer、避免过度 interface 装箱、减少临时对象。

## 内存分配

Go runtime 针对小对象和大对象有不同分配路径：

- 小对象通常从 per-P cache 中快速分配。
- 大对象直接走堆分配。
- runtime 会按大小规格管理内存块，减少碎片。

工程优化建议：

- 热路径减少临时对象。
- 使用 `strings.Builder` 或 `bytes.Buffer` 拼接大量字符串。
- 合理预估 slice/map 容量。
- 对可复用临时对象使用对象池，但不要滥用。

## Go 内存模型

Go 内存模型定义不同 goroutine 之间读写的可见性规则。核心是 happens-before。

常见 happens-before 关系：

- goroutine 创建前的写入，对新 goroutine 可见。
- channel 发送 happens-before 对应接收。
- close channel happens-before 接收到关闭事件。
- mutex unlock happens-before 后续 lock。
- atomic 操作提供特定同步语义。

面试重点：

不要依赖“看起来会先执行”的时间顺序。并发读写必须通过 channel、锁、atomic 等同步手段建立可见性。

## 常见调优方向

- 用 `pprof` 定位 CPU、内存、阻塞、锁竞争问题。
- 用 `go test -bench` 做基准测试。
- 用 `go test -race` 检查数据竞争。
- 用 context 和超时避免 goroutine 堆积。
- 控制并发数，避免把下游依赖打爆。

## 易错点

- goroutine 不是 OS 线程，但最终仍要由 OS 线程执行。
- `GOMAXPROCS` 控制同时执行 Go 代码的 P 数量，不是 goroutine 最大数量。
- GC 会自动回收内存，但不负责关闭文件、网络连接、锁等资源。
- 手动 `runtime.GC()` 通常不是常规优化手段。
- 内存逃逸不一定有问题，关键是是否在热路径造成大量分配。
