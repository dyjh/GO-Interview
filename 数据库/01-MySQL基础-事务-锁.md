# MySQL 基础、事务与锁

## MySQL 面试核心图谱

MySQL 面试通常围绕这些主题展开：

- 存储引擎：InnoDB、MyISAM 的区别。
- 事务：ACID、隔离级别、MVCC。
- 锁：表锁、行锁、记录锁、间隙锁、临键锁、意向锁。
- 并发问题：脏读、不可重复读、幻读、死锁。
- 实战优化：索引命中、锁范围、事务长度、读写分离、分库分表。

## 存储引擎

### InnoDB 和 MyISAM

| 对比项 | InnoDB | MyISAM |
| --- | --- | --- |
| 事务 | 支持 | 不支持 |
| 行锁 | 支持 | 不支持，主要是表锁 |
| 外键 | 支持 | 不支持 |
| 崩溃恢复 | 支持 redo/undo，恢复能力强 | 较弱 |
| 索引结构 | 聚簇索引 | 非聚簇索引 |
| 适用场景 | 绝大多数业务场景 | 读多写少、历史遗留场景 |

现在业务系统默认优先选择 InnoDB。

## 事务基础

### ACID

- Atomicity 原子性：事务内操作要么全部成功，要么全部失败。
- Consistency 一致性：事务前后数据满足约束和业务规则。
- Isolation 隔离性：并发事务之间互不干扰到指定程度。
- Durability 持久性：事务提交后数据不会因宕机轻易丢失。

实现相关：

- 原子性依赖 undo log。
- 持久性依赖 redo log。
- 隔离性依赖锁和 MVCC。
- 一致性由数据库约束、事务机制和应用逻辑共同保证。

### undo log、redo log 和 binlog

| 日志 | 主要作用 | 归属 |
| --- | --- | --- |
| undo log | 保存旧版本，用于回滚和 MVCC 读历史版本 | InnoDB |
| redo log | 记录页的物理修改，保证崩溃恢复和持久性 | InnoDB |
| binlog | 记录逻辑变更，用于主从复制、按时间点恢复和审计 | MySQL Server 层 |

容易混的点：

- undo log 解决“能不能回滚”和 MVCC 旧版本读取。
- redo log 解决“崩溃后已提交事务能不能恢复”。
- binlog 解决“复制和逻辑恢复”，不是 InnoDB 专属。
- MySQL 通过 redo log 和 binlog 的两阶段提交，降低崩溃恢复和主从复制时的不一致风险。

### 事务隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 说明 |
| --- | --- | --- | --- | --- |
| Read Uncommitted（读未提交） | 可能 | 可能 | 可能 | 基本不用 |
| Read Committed（读已提交） | 避免 | 可能 | 可能 | 每次快照读生成新的 Read View |
| Repeatable Read（可重复读） | 避免 | 避免 | 普通快照读基本避免；锁定读靠 next-key lock | MySQL InnoDB 默认级别 |
| Serializable（串行化） | 避免 | 避免 | 避免 | 并发性能差 |

注意：

- 标准 SQL 中 RR 仍可能幻读。
- InnoDB 在 RR 下通过 MVCC 处理快照读，通过 next-key lock 处理锁定读的幻读问题。
- 不能简单说“RR 完全解决幻读”，要区分快照读和当前读。

## MVCC

MVCC 是多版本并发控制，让读写并发更高。

核心组成：

- 隐藏字段：事务 ID、回滚指针等。
- undo log：保存旧版本数据。
- Read View：判断当前事务能看到哪些版本。

大概原理：

- 每行记录背后有事务 ID 和回滚指针。
- 更新数据时，不是简单覆盖旧值，而是生成新版本，并通过 undo log 保留旧版本。
- 多个版本会串成版本链。
- 普通 `SELECT` 做快照读时，会根据 Read View 判断版本链上哪个版本对当前事务可见。
- 如果最新版本不可见，就沿着 undo log 找更早的版本。

### 快照读和当前读

快照读：

```sql
SELECT * FROM user WHERE id = 1;
```

当前读：

```sql
SELECT * FROM user WHERE id = 1 FOR UPDATE;
UPDATE user SET name = 'a' WHERE id = 1;
```

快照读读历史可见版本，不加锁或少加锁；当前读读取最新数据，并可能加锁。

如果一个事务正在更新但还没提交，其他事务的快照读通常不会被阻塞，也不会读到未提交数据，而是读到当前 Read View 可见的旧版本。

RC 和 RR 的区别：

- RC：每次快照读都生成新的 Read View，所以同一事务内两次查询可能读到不同提交版本。
- RR：通常第一次快照读时创建 Read View，同一事务内复用，所以可以做到可重复读。

## MySQL 锁类型

### 表锁

锁整张表，粒度大，并发低。MyISAM 常用表锁；InnoDB 在某些 DDL、显式 `LOCK TABLES`、元数据锁场景也会涉及表级锁。

### 行锁

InnoDB 的行锁基于索引实现。只有通过索引定位到记录时，才能精确锁行。

如果 SQL 没有命中索引，InnoDB 可能扫描并锁住大量记录，看起来像“锁表”，但这不是传统意义上的锁升级。

### 记录锁 Record Lock

锁住索引记录本身。

常见场景：唯一索引等值命中某一行并执行当前读，例如：

```sql
SELECT * FROM user WHERE id = 1 FOR UPDATE;
```

### 间隙锁 Gap Lock

锁住索引记录之间的间隙，防止其他事务插入新记录。

常见触发场景：RR 隔离级别下，使用索引做范围条件的当前读，例如：

```sql
SELECT * FROM user WHERE age BETWEEN 10 AND 20 FOR UPDATE;
```

它会锁住范围内的间隙，防止其他事务插入满足条件的新记录。

### 临键锁 Next-Key Lock

记录锁 + 间隙锁，用于防止幻读。InnoDB RR 下范围查询的锁定读常见。

常见触发场景：RR 隔离级别下，范围查询命中索引并执行当前读，例如：

```sql
SELECT * FROM user WHERE id > 10 FOR UPDATE;
```

InnoDB 通常会锁住已有索引记录和记录前后的间隙。

### 意向锁

意向锁是表级锁，表示事务接下来想在表中某些行上加共享锁或排他锁。它用于协调表锁和行锁，不需要业务手动处理。

常见触发场景：事务准备给某些行加行锁时，InnoDB 会自动先在表上加意向锁。

- `SELECT ... LOCK IN SHARE MODE` 会先加意向共享锁 IS。
- `SELECT ... FOR UPDATE` 或 `UPDATE` 某些行时，会先加意向排他锁 IX。

## 悲观锁和乐观锁

### 悲观锁

认为并发冲突很可能发生，先加锁再操作。

```sql
BEGIN;
SELECT stock FROM goods WHERE id = 1 FOR UPDATE;
UPDATE goods SET stock = stock - 1 WHERE id = 1;
COMMIT;
```

适合：

- 冲突概率高。
- 对一致性要求高。
- 秒杀库存扣减等关键场景。

注意锁粒度和事务时长，否则吞吐会下降。

### 乐观锁

认为冲突较少，更新时检查版本号。

```sql
UPDATE goods
SET stock = stock - 1, version = version + 1
WHERE id = 1 AND version = 10 AND stock > 0;
```

适合：

- 读多写少。
- 冲突较低。
- 可重试的业务。

## 死锁

常见原因：

- 多个事务以不同顺序锁相同资源。
- 范围更新导致间隙锁冲突。
- SQL 没有命中索引，锁范围过大。
- 大事务持锁时间太长。

排查：

```sql
SHOW ENGINE INNODB STATUS;
```

优化方向：

- 固定资源访问顺序。
- 缩短事务。
- 确保更新条件命中索引。
- 减少范围锁。
- 捕获死锁错误并重试。

## MySQL 8.0 常见特性

面试可以简要提：

- 窗口函数。
- CTE 公共表表达式。
- JSON 能力增强。
- 默认字符集改进。
- 原子 DDL。
- 角色管理。
- InnoDB 和优化器增强。

不要只背特性名称，最好结合项目场景，例如用窗口函数做排名、分页统计等。

## 三大范式

- 第一范式：字段不可再分。
- 第二范式：非主键字段完全依赖主键，避免部分依赖。
- 第三范式：非主键字段不依赖其他非主键字段，避免传递依赖。

实际设计中会在规范化和性能之间权衡。高频读场景可适当冗余字段，但要有同步策略。

## 易错点总结

- InnoDB 行锁是基于索引的，没命中索引会扩大锁范围。
- InnoDB 一般不会像 SQL Server 那样因为锁太多自动升级为表锁。
- RR 不能简单说“完全解决幻读”，要区分快照读和当前读。
- 当前读会读最新数据并加锁，显式事务中锁会持续到提交或回滚。
- 快照读读的是 Read View 可见版本，不会读到其他事务未提交的数据。
- 间隙锁和临键锁主要出现在 RR 下范围条件的当前读中。
- redo log 和 binlog 不是一回事：redo 偏崩溃恢复，binlog 偏复制和逻辑恢复。

## 面试追问

### 为什么 InnoDB 使用 B+Tree？

B+Tree 层高低，范围查询友好，叶子节点有序并通过链表连接，磁盘 I/O 次数可控，适合数据库索引。

### RR 和 RC 的主要区别？

RC 每次快照读都生成新的 Read View，所以同一事务内两次查询可能读到不同提交版本。RR 通常在第一次快照读时创建 Read View，同一事务内可重复读。

### 如何降低锁冲突？

- 索引命中，缩小锁范围。
- 事务尽量短。
- 按固定顺序访问资源。
- 热点数据拆分。
- 库存类场景用 Redis 预扣、队列削峰、异步落库等架构手段。
