# MySQL 分库分表与读写分离

## 为什么需要拆分

当单库单表无法满足容量、吞吐或隔离需求时，需要考虑拆分。

常见信号：

- 单表数据量过大，索引层级、查询和 DDL 成本上升。
- 单库写入压力过大。
- 热点业务影响其他业务。
- 数据生命周期差异明显，冷热数据混在一起。
- 单机磁盘、CPU、连接数接近瓶颈。

拆分不是第一选择。优先做 SQL 优化、索引优化、归档/分区、缓存、读写分离，再评估分库分表。

## 垂直拆分

### 垂直分表

把一张宽表按字段拆开：

- 主表保存高频基础字段。
- 扩展表保存低频大字段。

适合：

- 表字段很多。
- 存在 text/blob/json 等大字段。
- 高频查询只需要少数字段。

### 垂直分库

按业务域拆库：

- 用户库。
- 订单库。
- 商品库。
- 支付库。

优点是业务隔离更清晰，缺点是跨库 join 和事务复杂。

## 水平拆分

把同一张表的数据按规则分散到多个表或库。

常见策略：

- 按 ID hash。
- 按用户 ID hash。
- 按时间范围。
- 按地域或业务线。
- 一致性哈希。

示例：

```text
order_00
order_01
...
order_15
```

路由：

```text
table = order_${user_id % 16}
```

## MySQL 分区

分区是把一张逻辑表按规则拆成多个物理分区，仍然属于同一个库、同一张表，对业务 SQL 基本透明。它更像单机内部的数据管理手段，不等于分库分表。

常见类型：

- RANGE 分区：按范围切分，最常见，例如按日期、月份、ID 范围分区。
- LIST 分区：按枚举值切分，例如地区、业务类型、状态。
- HASH/KEY 分区：按哈希打散，适合没有明显范围条件但希望数据分布更均匀的场景。

示例：

```sql
CREATE TABLE orders (
  id BIGINT NOT NULL,
  user_id BIGINT NOT NULL,
  created_at DATE NOT NULL,
  PRIMARY KEY (id, created_at)
)
PARTITION BY RANGE COLUMNS(created_at) (
  PARTITION p202601 VALUES LESS THAN ('2026-02-01'),
  PARTITION p202602 VALUES LESS THAN ('2026-03-01'),
  PARTITION pmax VALUES LESS THAN (MAXVALUE)
);
```

适合：

- 按时间增长的大表，例如订单、日志、流水、历史记录。
- 查询经常带分区字段，例如 `created_at`，可以触发分区裁剪，只扫描相关分区。
- 需要快速清理历史数据，例如直接 `DROP PARTITION` 删除某个月的数据。

注意：

- 分区不能突破单机 CPU、内存、磁盘和连接数瓶颈。
- 查询如果不带分区字段，仍可能扫描多个分区。
- MySQL 分区表的唯一键通常需要包含分区键，否则无法保证全局唯一性。
- 最新分区可能形成热点，按时间分区时要特别注意写入集中问题。

分区和分库分表的区别：

| 对比 | 分区 | 分库分表 |
| --- | --- | --- |
| 位置 | 同一个库、同一张逻辑表内部 | 多张表或多个库 |
| 业务感知 | 低，SQL 基本不变 | 高，需要路由或中间件 |
| 扩展能力 | 主要改善管理和部分查询 | 可以扩展容量和吞吐 |
| 复杂度 | 相对较低 | 明显更高 |
## 分片键选择

分片键非常关键，选错后很难调整。

好的分片键：

- 高频查询条件中稳定出现。
- 分布均匀，避免热点。
- 不频繁变更。
- 能减少跨分片查询。

常见选择：

- 用户维度系统：`user_id`。
- 订单系统：如果多按用户查订单，用 `user_id`；如果多按订单查，用订单 ID 中嵌入分片信息。
- 时间序列数据：按时间分表，但要注意最新分片热点。

## 分库分表后的问题

### 跨库 join

解决方向：

- 业务层组装。
- 字段冗余。
- 建宽表或搜索索引。
- 数据同步到 OLAP/ES。

### 分布式事务

解决方向：

- 尽量避免强一致跨库事务。
- 本地事务 + 消息最终一致。
- TCC、Saga。
- Outbox pattern。

### 全局唯一 ID

方案：

- Snowflake 类 ID。
- 数据库号段。
- Redis/etcd 发号。
- UUID/ULID。

业务高并发系统常用趋势递增 ID，兼顾唯一性和索引写入性能。

### 跨分片分页和排序

跨分片排序需要各分片查出局部结果，再归并排序。成本高，应尽量通过查询条件定位分片，或引入搜索/分析系统。

### 扩容迁移

普通取模扩容会导致大量数据迁移。可考虑：

- 预分片。
- 一致性哈希。
- 双写迁移。
- 后台迁移 + 校验 + 切流。

## 分库分表中间件与替代方案

MyCat 是较早期的分库分表中间件，现在更常见的选择会按场景区分。

| 方案 | 适合场景 | 说明 |
| --- | --- | --- |
| ShardingSphere | Java 项目做分库分表、读写分离 | 有 JDBC 和 Proxy 两种形态，生态更活跃。 |
| Vitess | 大规模 MySQL 分片治理 | 常用于云原生 MySQL 分片、路由、拓扑管理。 |
| TiDB / 分布式 SQL | 能接受换数据库，希望少维护分片逻辑 | 兼容 MySQL 协议，底层自动水平扩展。 |
| ProxySQL | 主要做读写分离、连接池、SQL 路由 | 不是完整分库分表方案，更偏 MySQL Proxy。 |
| MyCat | 旧系统或已有 MyCat 架构 | 新项目一般不优先选。 |

中间件的价值：

- 对业务代码侵入较小。
- 集中管理路由规则。
- 支持读写分离和部分 SQL 透明化。

注意：

- 复杂 SQL、跨分片 Join、聚合、排序分页仍然要谨慎设计。
- 增加中间层后，排障链路会变长。
- 分库分表的核心问题不会因为用了中间件就消失，分片键、事务、扩容、数据一致性仍要提前设计。
## 读写分离

读写分离通过主库写、从库读来扩展读能力。

基本流程：

1. 写请求发送到主库。
2. 主库通过 binlog 将变更复制到从库。
3. 读请求按策略路由到从库。

常见复制方式：

- 异步复制：性能好，但有延迟和丢数据风险。
- 半同步复制：主库至少等待一个从库确认，可靠性更高但延迟增加。
- 组复制/MGR：更强的一致性和高可用能力，但复杂度更高。

### 主从延迟

主从延迟可能导致“刚写完读不到”。

解决方案：

- 写后读主库。
- 根据用户会话短时间读主。
- 关键读请求强制走主库。
- 监控复制延迟，延迟过大时摘除从库。
- 业务允许最终一致时提示处理中。

## 分库分表与缓存

分片后缓存 key 要包含分片或业务唯一标识：

```text
order:{order_id}
user_orders:{user_id}:{page}
```

注意缓存失效和数据库分片路由保持一致，避免跨库数据错读。

## 什么时候不要分库分表

- 数据量还不大，只是 SQL 没优化。
- 没有明确分片键。
- 大量跨用户、跨维度查询。
- 团队缺乏运维和迁移能力。
- 业务仍快速变化，模型未稳定。

分库分表会显著增加复杂度，应当是容量和吞吐瓶颈明确后的方案。

## 策略选择总结

一般推荐顺序：先优化 SQL 和索引，再考虑归档/分区、缓存、读写分离，最后才是分库分表。查询效率提升不能只看“用了什么策略”，还要看是否命中索引、缓存、分区键或分片键。

| 场景 | 优先策略 | 单次查询延迟提升 | 整体读吞吐提升 | 说明 |
| --- | --- | --- | --- | --- |
| SQL 慢但数据量不大 | SQL 优化、索引优化 | 高 | 中 | 最直接，能把全表扫描变成索引定位时提升最明显。 |
| 宽表字段多、大字段多 | 垂直分表 | 中 | 中 | 高频字段和低频大字段拆开，减少单行 IO。 |
| 不同业务边界清晰 | 垂直分库 | 低到中 | 中 | 主要提升业务隔离和独立扩展，单次查询不一定明显变快。 |
| 历史数据多，常按时间查询或清理 | 分区、归档 | 中到高 | 中 | 查询命中分区键或热表变小后效果明显；历史查询可能变慢。 |
| 读多写少，读压力大 | 读写分离、缓存 | 缓存很高；读写分离低 | 高到很高 | 缓存命中最快；读写分离主要提高读并发，要注意主从延迟。 |
| 单表容量和写入压力都很大，且有稳定分片键 | 水平分库分表 | 中到高 | 高 | 能路由到单分片时效果好；跨分片查询可能更慢。 |
| 热点数据压力大 | 缓存、队列削峰、热点拆分 | 低到中 | 中到高 | 重点是降低热点和峰值压力，不一定提升普通单次查询速度。 |
| 多维度搜索、报表、聚合分析 | ES、OLAP、宽表 | 高 | 高 | 适合搜索、聚合、报表；不适合替代强一致的主库事务查询。 |
| 强一致事务很多 | 尽量单库内解决，谨慎拆分 | 低 | 低到中 | 目标是降低一致性复杂度，不是优先追求查询提速。 |

简单记忆：索引优化和缓存最容易直接提升查询速度；读写分离主要提升读吞吐；分区和分库分表只有在查询能命中分区键或分片键时效果才明显。
## 面试追问

### 分库分表后如何查一个用户的订单？

如果按 `user_id` 分片，直接路由到对应分片查询。如果按 `order_id` 分片，则需要维护 `user_id -> order_id` 索引关系或构建用户订单列表表/搜索索引。

### 如何平滑扩容？

可使用双写、增量同步、历史数据迁移、校验、灰度切流。扩容前最好预留足够分片，减少频繁迁移。

### 如何保证最终一致？

本地事务提交业务数据和消息记录，然后异步投递消息；消费者幂等处理；失败重试；定时补偿；关键链路做对账。
