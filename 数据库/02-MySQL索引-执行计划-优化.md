# MySQL 索引、执行计划与优化

## 核心定位

MySQL 查询优化不要只理解成“加索引”。更完整的思路是：先确认 SQL 要查什么，再看过滤、排序、分组、Join、回表这些成本，最后用 `EXPLAIN` 验证执行计划。

面试回答时重点说清楚：

- 查询慢在哪里：扫描行数多、排序重、回表多，还是 Join / 聚合成本高。
- 索引能解决什么：减少扫描、减少排序、减少回表。
- SQL 写法有没有破坏索引使用。
- 执行计划里实际用了哪个索引、扫描了多少行、有没有临时表或 filesort。

## 索引基础

### 索引的作用和代价

索引是帮助 MySQL 快速定位数据的数据结构，InnoDB 常见索引底层是 B+Tree。

它的好处是提升查询效率，尤其是减少全表扫描、排序和回表成本。但索引不是越多越好，它也有代价：

- 占用磁盘空间。
- 写入、更新、删除时需要维护索引。
- 索引过多会增加优化器选择成本，也可能影响写入性能。

### B-Tree、B+Tree 和普通树的区别

数据库索引常说的 `BTREE`，更准确地说在 InnoDB 中通常是 B+Tree。它不是普通二叉树，而是一种更适合磁盘页存储的多路平衡树。

| 结构 | 特点 | 对数据库索引的影响 |
| --- | --- | --- |
| 普通二叉搜索树 | 每个节点最多 2 个子节点，数据量大时树高容易变高 | 查找可能需要很多层，随机 I/O 次数多，不适合磁盘索引 |
| B-Tree | 多路平衡树，非叶子节点和叶子节点都可以存 key 和数据指针 | 树高比二叉树低，但范围扫描时需要在多层节点之间移动 |
| B+Tree | 非叶子节点主要存 key 和子节点指针，真实数据或行指针集中在叶子节点；叶子节点有序并通过链表连接 | 扇出更大、树高更低，范围查询和顺序扫描更友好 |

现在 MySQL / InnoDB 更常用 B+Tree，核心原因是：

- 非叶子节点不存完整行数据，同一个数据页能放更多 key，树更矮，磁盘 I/O 更少。
- 所有数据都在叶子节点，查询路径更稳定。
- 叶子节点天然有序并相互连接，`BETWEEN`、`ORDER BY`、范围分页这类查询可以顺着叶子节点扫描。
- InnoDB 的聚簇索引和二级索引都能很好映射到 B+Tree：主键索引叶子节点存整行数据，二级索引叶子节点存索引列和主键值。

### 常见索引类型

| 类型 | 说明 |
| --- | --- |
| 主键索引 | 唯一且非空，一张表通常只有一个主键。 |
| 唯一索引 | 保证列值唯一，可以有多个。 |
| 普通索引 | 最常见，用于加速普通查询。 |
| 联合索引 | 多个列组成一个索引，设计时要考虑列顺序。 |
| 全文索引 | 适合特定文本检索，但复杂搜索通常用专门搜索系统。 |

### 聚簇索引、二级索引和回表

InnoDB 表数据按主键组织，主键索引的叶子节点存储完整行数据，这就是聚簇索引。

二级索引的叶子节点通常存储：

- 二级索引列值。
- 主键值。

通过二级索引查到主键后，再回到聚簇索引查询完整行数据，这个过程叫回表。

### 覆盖索引

如果查询所需字段都在同一个索引中，就不需要回表，这叫覆盖索引。

```sql
CREATE INDEX idx_user_status_created ON user(status, created_at, id);

SELECT id, status, created_at
FROM user
WHERE status = 1
ORDER BY created_at
LIMIT 20;
```

这个查询可能使用覆盖索引，因为 `SELECT`、`WHERE`、`ORDER BY` 需要的字段都在索引里。

## 索引设计

### 联合索引与最左前缀

联合索引 `(a, b, c)` 通常可以高效支持：

- `WHERE a = ?`
- `WHERE a = ? AND b = ?`
- `WHERE a = ? AND b = ? AND c = ?`
- `WHERE a = ? ORDER BY b`

通常不能高效支持：

- `WHERE b = ?`
- `WHERE c = ?`

原因是联合索引要从最左列开始匹配。

范围条件后面的列，通常难以继续用于精确定位：

```sql
WHERE a = ? AND b > ? AND c = ?
```

这里 `a` 和 `b` 可用于索引范围定位，`c` 可能只能做索引条件过滤，具体要看 MySQL 版本和执行计划。

### 索引选择性

选择性越高，索引越有价值。选择性可以粗略理解为：字段值越分散，过滤效果越好。

```sql
SELECT COUNT(DISTINCT status) / COUNT(*) FROM orders;
```

像性别、状态位这类低选择性字段，不一定适合单独建索引。但它们可以作为联合索引的一部分，服务于具体查询。

### 单列索引和联合索引

多个单列索引不等于一个合适的联合索引。MySQL 有 `index merge` 能力，但很多场景下不如专门设计的联合索引稳定。

设计联合索引时要考虑：

- 查询是否高频。
- 等值条件有哪些。
- 范围条件在哪里。
- 是否需要排序和分组。
- 是否能做覆盖索引。

### 索引下推 ICP

Index Condition Pushdown 可以在存储引擎层用索引字段先过滤部分条件，减少回表次数。

例如联合索引 `(a, b)`：

```sql
SELECT * FROM t WHERE a > 10 AND b = 3;
```

虽然 `b` 不一定参与范围定位，但可以在扫描索引时先过滤 `b`，减少回表。

## 常见 SQL 写法

### 容易导致索引失效的写法

在索引列上使用函数：

```sql
WHERE DATE(created_at) = '2026-01-01'
```

建议改成范围查询：

```sql
WHERE created_at >= '2026-01-01'
  AND created_at < '2026-01-02'
```

隐式类型转换：

```sql
WHERE phone = 13800138000
```

如果 `phone` 是 `varchar`，应写成字符串：

```sql
WHERE phone = '13800138000'
```

左模糊匹配：

```sql
WHERE name LIKE '%abc'
```

对索引列做表达式计算：

```sql
WHERE amount + 1 > 100
```

`OR` 条件中部分字段没有索引，也可能导致全表扫描。

### GROUP BY 和 HAVING

`GROUP BY` 用来分组统计，`HAVING` 用来过滤分组后的结果。

简单理解执行顺序：

```text
WHERE 先过滤明细行
GROUP BY 再分组聚合
HAVING 最后过滤聚合结果
```

例子：统计每个用户已支付订单数和总金额，只保留订单数不少于 3 的用户。

```sql
SELECT user_id,
       COUNT(*) AS order_count,
       SUM(amount) AS total_amount
FROM orders
WHERE status = 'paid'
GROUP BY user_id
HAVING COUNT(*) >= 3
ORDER BY total_amount DESC;
```

`WHERE` 和 `HAVING` 的区别：

| 对比 | WHERE | HAVING |
| --- | --- | --- |
| 执行位置 | 分组前 | 分组后 |
| 过滤对象 | 原始行 | 分组后的结果 |
| 能否使用聚合函数 | 一般不能 | 可以，例如 `COUNT(*)`、`SUM(amount)` |
| 常见用途 | 先缩小扫描范围 | 过滤统计结果 |

注意点：

- 能写在 `WHERE` 的条件不要放到 `HAVING`，先过滤明细行可以减少参与分组的数据量。
- `SELECT` 中非聚合字段通常要出现在 `GROUP BY` 中，开启 `ONLY_FULL_GROUP_BY` 时尤其要注意。
- `GROUP BY` 字段如果能匹配联合索引顺序，可能减少临时表和排序成本。
- 大表高频实时聚合成本较高，可以考虑异步汇总表、宽表、缓存或 OLAP。

### Join 优化

Join 常见优化方向：

- 小表驱动大表。
- Join 字段类型和字符集保持一致。
- Join 字段建立合适索引。
- 避免一次 Join 过多表。
- 必要时拆成多次查询，或通过字段冗余、宽表降低 Join 成本。

### 排序优化

如果排序字段能利用索引，就可以减少 `filesort`。

```sql
CREATE INDEX idx_status_created ON orders(status, created_at);

SELECT *
FROM orders
WHERE status = 1
ORDER BY created_at
LIMIT 20;
```

如果 `ORDER BY` 和 `WHERE` 的顺序与索引不匹配，可能无法使用索引排序。

## 执行计划 EXPLAIN

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 1;
```

常看字段：

| 字段 | 关注点 |
| --- | --- |
| `type` | 访问类型，常见从好到差有 `const`、`eq_ref`、`ref`、`range`、`index`、`ALL`。 |
| `possible_keys` | 可能使用的索引。 |
| `key` | 实际使用的索引。 |
| `rows` | 预计扫描行数，越少通常越好。 |
| `Extra` | 额外信息，例如 `Using index`、`Using where`、`Using temporary`、`Using filesort`。 |

常见判断：

- `Using index` 通常表示使用了覆盖索引。
- `Using filesort` 不一定绝对坏，但大结果集排序要重点关注。
- `Using temporary` 常见于分组、排序、去重，数据量大时要小心。
- `type = ALL` 通常表示全表扫描，需要结合数据量和业务场景判断是否要优化。

## 慢 SQL 优化流程

排查慢 SQL 可以按这个顺序：

1. 确认业务需求，避免查无用字段和无边界查询。
2. 查看执行计划，确认扫描行数和索引使用情况。
3. 分析 `WHERE`、`JOIN`、`ORDER BY`、`GROUP BY`，设计合适的联合索引。
4. 让高选择性字段尽量靠前，但要结合等值、范围、排序和分组需求。
5. 减少回表，必要时使用覆盖索引。
6. 大分页改为游标/seek 查询。
7. 大表统计和聚合考虑异步汇总、宽表、缓存或 OLAP。

面试里可以这样答：

> 先看业务和 SQL 写法，再看执行计划，确认是否走索引、扫描多少行、是否出现临时表或 filesort。然后围绕过滤、排序、分组和回表成本设计联合索引，必要时改 SQL 或做异步汇总。

## 面试追问

### 为什么不建议 SELECT *？

- 可能导致无法使用覆盖索引。
- 读取无用列增加 I/O 和网络传输。
- 表结构变化可能影响调用方。
- 大字段如 `text`、`blob` 会显著增加成本。

### 回表是什么？

通过二级索引查到主键后，再回到聚簇索引查询完整行数据的过程。覆盖索引可以避免回表。

### 索引越多越好吗？

不是。索引会占空间，影响写入，增加优化器选择成本。应围绕高频查询和业务路径设计索引。

### 如何判断该不该建索引？

- 查询是否高频。
- 过滤条件选择性是否足够。
- 是否能减少排序、分组、回表或扫描行数。
- 写入成本是否可接受。
- 是否已有联合索引可以复用。

### GROUP BY 慢怎么优化？

优先用 `WHERE` 减少参与分组的数据量，再看 `GROUP BY` 字段是否能利用索引。大表高频统计不要每次实时聚合，可以考虑汇总表、缓存或 OLAP。
